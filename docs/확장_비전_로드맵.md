# 그때그때 — 확장·비전 로드맵 v0.2

> **이 문서의 위치**: 확정된 MVP(`그때그때_PRD_v1.0.md`) **이후**의 확장 방향을 담는다.
> ⚠️ 이 문서는 MVP를 대체하거나 수정하지 않는다. **모든 확장은 코어(사진·영상)가 습관으로 붙는다는 게 지표로 증명된 뒤** 착수한다.
> v0.2는 기획 초안(v0.1, 2026-07-02)을 레포에 흡수하며 구현팀 분석 보정 3건(§5 리스크 재평가, §6 내부 순서, §8 선결 조건)을 반영한 판이다.

---

## 1. 비전 한 줄

> **그때그때는 "미루면 쌓이는 것"을 매일 정한 시간에 그때그때 처리하게 만드는 습관 엔진이다.**

지금은 그 엔진을 "사진·영상 정리"에 꽂았을 뿐, 브랜드 이름부터가 특정 기능이 아니라 **"미루지 않기"라는 철학**을 담고 있다.

## 2. 핵심 자산 = 엔진 (도메인은 부품)

그때그때의 재사용 가능한 엔진은 3박자다:

1. **알림** — 사용자가 정한 시간에 트리거.
2. **백로그 처리** — 그동안 쌓인 것을 작은 더미로 빠르게 처리.
3. **streak** — 연속 실행으로 습관화.

이 엔진은 도메인 무관하다. 따라서 확장 = **"이 엔진을 어떤 도메인에 꽂을 수 있는가"**의 문제다.

## 3. 도메인 선정 3관문

아무 도메인이나 후보가 아니다. 아래 3개를 **모두** 통과해야 한다:

| 관문 | 질문 | 통과 못하면 |
|------|------|-------------|
| ① 접근성 | 앱이 그 데이터에 **합법적으로 접근** 가능한가? | 플랫폼이 막으면 구현 불가 (아무리 좋아도 탈락) |
| ② 백로그성 | 처리 안 하면 **쌓이는** 성질인가? | 안 쌓이면 "그때그때" 할 이유가 없음 |
| ③ 그때그때 우위 | 몰아서보다 **매일 처리가 확실히 나은가**? | 우위 없으면 알림이 잔소리가 됨 |

## 4. 도메인 접근성 지도

| 도메인 | ① 접근 | ② 백로그 | ③ 우위 | 판정 | 근거 |
|--------|--------|----------|--------|------|------|
| 사진·영상 | ✅ | ✅ | ✅ | **가능(현재 MVP)** | PhotoKit / MediaStore로 접근 |
| 스크린샷·중복 | ✅ | ✅ | ✅ | **가능** | 사진 권한 그대로 재사용 |
| 저장공간(대용량·오래된 미디어) | ✅ | ✅ | △ | **가능** | 같은 권한, "비우기" 니즈 |
| 파일·다운로드 | △ | ✅ | △ | 제한적 | Android는 가능, iOS는 파일 접근 제약 |
| 메일 | △ | ✅ | △ | 제한적 | 계정 연동·API 복잡, 별 제품에 가까움 |
| 메시지·SMS | ❌ | ✅ | ✅ | **불가** | iOS 서드파티 읽기 API 없음 / Android는 기본 SMS 앱만(정책 제한) |
| 카카오톡 | ❌ | ✅ | ✅ | **불가** | 외부 앱이 개인 대화를 읽는 공개 API 없음 |

**핵심 교훈**: "쌓여서 아픈 것"과 "앱이 접근 가능한 것"은 다르다. 메시지·카톡은 백로그성·우위는 만점이지만 **①접근성에서 탈락** → 만들고 싶어도 문이 잠겨 있다. (iOS 앨범에서 배운 교훈의 반복)

## 5. 단계별 확장 로드맵

| Phase | 범위 | 착수 게이트 | 리스크 |
|-------|------|-------------|--------|
| **0 (현재 MVP)** | 사진·영상 정리 | — (구현 완료, 실기기 검증 대기) | — |
| **1 · 근시일 (추천)** | **미디어 깊이 확장** — 스크린샷 정리 / 중복 사진 / 저장공간 비우기 | Phase 0의 **North Star·D7 리텐션 증명** + **PRD v1.1 개정**(§8-1) | **낮음~중간** (같은 권한이지만 삭제는 파괴적 작업 — §5.1) |
| **2 · 중기** | 파일·다운로드 정리 / (가능 시) 메일 정리 | Phase 1 리텐션 + 접근성 개별 검증 | 중간 (플랫폼 제약·복잡도) |
| **3 · 장기 비전** | "그때그때 엔진" 모듈형 플랫폼 (접근 가능 도메인만 플러그인화) | Phase 1~2로 "엔진이 도메인 무관하게 먹힌다" 입증 | 높음 (조기 착수 시 코어 붕괴) |

> **핵심 원칙**: 수평 확장은 **코어를 증명한 뒤**. 각 Phase는 이전 Phase의 리텐션이 증명돼야 다음으로 간다. 하나가 습관으로 붙는지도 모르는 채 도메인을 늘리면 리스크가 곱해진다.

### 5.1 Phase 1 리스크 재평가 (v0.2 보정)

초안의 "리스크 낮음(같은 권한, 검증된 UX)"은 절반만 맞다. **권한은 같지만 작업의 성격이 다르다**:

- MVP의 배정은 **비파괴**(iOS 앨범 태깅 / Android 폴더 이동 — 원본 보존).
- Phase 1의 중복정리·저장공간 비우기는 **파괴적(삭제)**. iOS는 삭제마다 시스템 확인 다이얼로그가 강제되고, Android도 휴지통/삭제 요청 시 사용자 동의 창이 뜬다.
- 따라서 "탭 한 번에 던진다"는 검증된 UX가 삭제에는 그대로 이식되지 않는다. 삭제용 확정 흐름(일괄 확인·되돌리기 유예 등)을 별도 설계해야 한다.

## 6. Phase 1 상세 (근시일 추천 = "미디어 깊이")

같은 사진 권한 안에서 가치를 넓히는 확장. 셋 다 3관문을 통과하며 **엔진(알림·streak)을 그대로 재사용**한다.

**내부 착수 순서 (v0.2 보정 — 난이도 오름차순):**

| 순서 | 항목 | 난이도 | 근거 |
|------|------|--------|------|
| 1 | **스크린샷 정리** | 쉬움 | iOS(`PHAssetMediaSubtype.photoScreenshot`)·Android(시스템 스크린샷 버킷) 모두 OS가 스크린샷을 구분해 줌 — 필터만 추가하면 기존 정리 루프 재사용 |
| 2 | **저장공간 비우기** | 중간 | 크기·날짜 정렬은 쉬우나 **삭제 UX**(§5.1)가 신규 |
| 3 | **중복·유사 사진** | 어려움 | 백엔드 없음 원칙상 **온디바이스 판별** 필수. 완전 동일/지각 해시(이미지를 작은 지문으로 요약해 비교) 수준은 현실적이나, 연사·유사 컷 그룹핑은 난이도 한 단계 위 |

> **수요 신호 (2026-07-06):** 첫 실기기 사용성 테스트(소유자, 16k장 라이브러리)에서 정리 루프 도중 "삭제 기능이 필요하다"는 니즈가 실제로 나옴. 골든 룰 2에 따라 MVP에선 미구현(로드맵 기록만 결정). Phase 1 착수 심사 시 이 신호를 수요 근거로 참조할 것. 단 §5.1 파괴적 삭제 리스크와 "스와이프 방향 3개 소진 → 오조작=오삭제" 제스처 충돌 문제는 설계 단계 필수 검토 항목.

## 7. 주의 / 배제

- **조기 확장 함정**: Phase 0 리텐션 미검증 상태에서 Phase 1+로 점프 금지.
- **차단 도메인 배제**: 메시지·카톡 등 플랫폼이 막은 도메인은 로드맵에서 제외(재검토는 정책 변화 시).
- **Path C(사용자 정의 도메인) 주의**: 앱이 못 읽는 걸 "사용자가 직접 등록"하게 하면 습관/할일 앱(Finch·Streaks 등)과 경쟁 구도가 되고, "자동으로 쌓인 걸 정리해준다"는 차별점이 사라진다. → 신중히.
- **조기 추상화 금지**: Phase 3의 "엔진 플러그인화"를 미리 준비한다는 명목으로 지금 코어를 리팩터링하지 말 것. 추상화는 두 번째 도메인이 실제로 착수될 때 그 요구에 맞춰 뽑는다.

## 8. 착수 선결 조건 (v0.2 보정 — 게이트의 게이트)

1. **PRD v1.1 개정이 Phase 1 착수의 선행 절차다.** 현행 PRD 8절 Out of Scope는 **삭제·중복정리**를 명시적으로 범위 밖에 둔다. Phase 1의 "중복 사진"·"저장공간 비우기"가 정면 충돌하므로, 착수 시점에 PRD를 먼저 개정하지 않으면 "PRD가 기준"인 빌드 하네스가 구현을 거부한다.
2. **분석 실연동이 로드맵 전체의 선결 조건이다.** 모든 Phase의 착수 게이트가 "North Star·D7 리텐션 증명"인데, 측정이 안 되면 게이트 통과 여부를 영원히 알 수 없다. → Firebase Analytics 실연동은 "나중에"가 아니라 **출시 전 필수**.
