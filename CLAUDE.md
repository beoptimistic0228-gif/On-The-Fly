2. **MVP 범위를 사수하라.** PRD 8절의 Out of Scope(삭제·중복정리 / 태그 / AI 그룹화 / 중첩폴더 / 다중배정 / 통계 / 클라우드·계정)는 내가 명시적으로 범위를 바꾸기 전엔 **구현 금지**. 범위 밖 요청엔 "이건 MVP 밖이라 핵심 사전이 흐려진다"고 말하라.
   - **예외(2026-07-06 사용자 승인, D5)**: **정리 루프 내 단건 삭제**는 범위 편입. 중복정리·일괄 삭제는 여전히 금지. 상세: `_workspace/00_decisions.md` D5.
3. **핵심 사전 우선.** 온보딩 → 홈 → 정리(스와이프) → 완료, 이 단일 사전을 end-to-end로 먼저 완성한다.
4. **큰 변경은 plan mode로.** 아키텍처·다수 파일 변경 전에 계획을 먼저 보여주고, 승인 후 실행하라.

## 핵심 루프 (이 흐름이 MVP의 전부)
1. **매일 로컬 알림** (사용자가 정한 시간).
2. **미분류 자산 로드** — 단, "미분류" 판단은 **OS 상태가 아니라 앱이 기록한 처리 자산 ID 기준**이다. (iOS 앨범은 태그라 정리 후에도 원본이 타임라인에 남는다 → OS로 판단하면 안 됨.) 마지막 처리 이후 신규 자산만 노출.
3. **스와이프로 앨범에 배정** (기존 선택 또는 인라인 신규 생성).
4. **시스템 라이브러리에 반영** — iOS: 앨범 태깅(PhotoKit) / Android: 폴더 이동(MediaStore·Scoped Storage).
5. **처리 자산 ID 기록**(중복 방지) → 다음 날 같은 자산이 다시 안 뜨게.

## 기술 가드레일 (PRD 확정 사항)
- **백엔드 없음.** 온디바이스 + 로컬 저장(처리한 자산 ID)만. 서버·계정·로그인 만들지 마라.
- **프라이버시 = 핵심.** 사진·영상을 절대 폰 밖으로 내보내지 마라. 참조(ID)만 저장. 이건 마케팅·프라이버시 약속이기도 하니 온보딩·설정에서 노출.
- **데이터 모델(로컬)**: `ProcessedAsset(id, processedAt, albumId)` + `Album`(로컬 참조/캐시). 원본은 저장 안 함.
- **플랫폼 API**: iOS = PhotoKit(전체 접근 필요, 앨범은 화면/태깅), Android = Scoped Storage/MediaStore(파일 이동 시 사용자 동의 창 → 배치 승인 + 사전 안내).
- **iOS 성취감 리스크**: 앨범=태그라 원본 타임라인이 그대로 → "정리됐다" 느낌이 약함. 완료 화면·streak 시각화로 성취감을 보완하라(PRD 리스크 항목).
- **알림**: 로컬 알림(서버 푸시 없음).
- **수익화**: 광고 = 설치 7일 후, "완료" 화면 **뒤에서 세션당 1회만**(정리 흐름 중간 삽입 금지). 광고 제거 IAP(일회성). 결제·광고·분석 전부 SDK/스토어 처리, 서버 없음.
- **지표**: North Star(주간 정리 완료자)·D7 리텐션·세션당 미분류 감소를 산출할 분석 이벤트(F-12)를 반드시 심어라.
- **요구사항 ID**: 기능 요구사항은 PRD 6절에 F-01 ~ F-13으로 정의됨. D5 편입분(단건 삭제)은 F-14 계열(`_workspace/06_architect_delete.md`). 작업 시 해당 ID로 참조.

## 기술 스택
- **Flutter 확정** (2026-07-02, 사용자 결정). 이유: 사진 라이브러리 생태계(photo_manager 등)가 가장 성숙 — 이 앱의 핵심인 iOS PhotoKit/Android MediaStore 연동과 썸네일 그리드 성능에 가장 유리.
- 권장 조합(설계 단계에서 근거와 함께 확정): 상태관리 Riverpod, 로컬 DB Drift(SQLite), 사진 photo_manager, 알림 flutter_local_notifications, 라우팅 go_router.
