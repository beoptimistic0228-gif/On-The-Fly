## 1. 개요 (Summary)

- **한 줄 요약**: 그때그때는 **바쁜 1인 크리에이터**가 **미분류 사진·영상을 나중에 몰아서 정리하는 고통**을, **매일 정한 시간에 '아직 안 담은 것'만 스와이프로 이벤트 앨범에 배정**하게 만들어 해결한다.
- **배경**: 사진·영상은 매일 쌓이지만 정리는 계속 미뤄진다. 결국 백업·발행 시점에 거대한 미분류 더미를 한 장씩 확인하며 갈라놓아야 하고, 이 반복 노동이 지겨워 더 미루는 악순환이 생긴다. 그때그때는 그 더미를 **매일 작게 쪼개 "짧은 잡일"로** 바꿔 이 악순환을 끊는다.

## 2. 문제 정의 (Problem)

- **타겟 유저**: 여러 플랫폼(YouTube·블로그·Threads 등)에 콘텐츠를 발행하는 **1인 크리에이터·브랜드 운영자**. (앵커 페르소나 = 낙관 본인 → 매일 직접 사용하며 검증 가능)
- **페인포인트**: 미분류된 큰 사진/영상 더미를 **한 장씩 확인하며 의미(이벤트) 단위로 갈라놓는 수작업**이 지겨워 계속 미룬다.
- **핵심 원인**: 병목은 "기억력"이 아니다. 사용자는 사진만 봐도 뭔지 안다. 진짜 병목은 **처리량(더미 크기) × 한 장씩 손대는 반복 노동**이다. 더미가 클수록 "지치는 마라톤"이 되어 극도로 미뤄진다.
- **현재의 대안**: 방치 → SSD 백업 시점에 몰아서 수동 분류. 기본 갤러리는 "미분류분만 골라주기 / 의미 단위로 묶어주기 / 정리하라고 알려주기"를 하지 않는다.
- **왜 지금**: 크리에이터가 다루는 사진·영상량이 폭증. 온디바이스만으로 앨범 배정·로컬 알림이 충분히 가능해 서버 없이도 구현 가능.
- **근거**: 앵커 사용자 본인의 반복 경험 + 주변 지인 관찰. (정량 검증은 오픈 이슈 → 12절)

## 3. 목표 & 성공 지표 (Goals & Metrics)

- **제품 목표**: "몰아서 정리"를 "매일 그때그때"로 바꿔 **정리를 습관화**한다.
- **North Star Metric**: **주간 정리 완료 사용자 수** — 한 주에 1회 이상 정리 세션을 완료한 사용자. (다운로드 수보다 정직하게 "매일 쓰는 도구가 됐나"를 대표)
- **보조 지표(KPI)**:
  - KPI 1 — **D7 리텐션**(가입 7일 뒤 재방문율): 습관 형성의 핵심 검증.
  - KPI 2 — **세션당 정리 매수 · 미분류 잔량 감소**: 정리가 실제로 일어나는지.

## 4. 솔루션 개요 (Solution)

- **핵심 가치 제안**: 하루 한 번, 미분류 사진·영상을 스와이프로 앨범에 담는다.
- **차별점**:
  - 기본 갤러리 = **창고**(알려주지도, 미분류분만 골라주지도 않음). 그때그때 = **매일의 정리 루틴**(미분류분만 큐 추출 → 알림 → 스와이프 한 번).
  - **서버 없음 = 강력한 프라이버시**. 사진·영상은 폰 밖으로 나가지 않는다. → 핵심 마케팅 메시지.
- **핵심 컨셉 (동작 원리)**:
  1. **매일 알림** (사용자가 설정한 시간).
  2. **미분류 큐 로드** — "지난 처리 이후 새 사진·영상"만. 미분류 기준은 OS에 묻지 않고 **앱이 처리한 자산 ID를 직접 기록**해 판단(iOS 앨범=태깅이라 타임라인에 남는 문제와 무관하게 큐가 정확).
  3. **스와이프로 배정** — 기존 앨범 선택 또는 새 앨범 생성(평면/단일 계층).
  4. **시스템 앨범 반영** — iOS: 앨범 태깅 / Android: 폴더 이동.
  5. **완료 & 큐 갱신** — 처리한 자산 ID 기록(중복 방지) → 다음 날 새 큐로 자동 반복.

## 5. 사용자 스토리 & 유즈케이스 (User Stories)

- 크리에이터로서, 나중에 빨리 찾기 위해, **촬영 당일 사진·영상을 이벤트 앨범에 담고** 싶다.
- 바쁜 사용자로서, 미루지 않기 위해, **매일 정한 시간에 정리 알림**을 받고 싶다.
- 백업하는 사람으로서, **앨범 단위로 내보내기** 위해, 사진이 미리 이벤트별로 분류돼 있길 원한다.

## 5.5. 핵심 화면 & 화면 흐름 (Screens & Flow)

- **핵심 화면 목록 (6개)**:
  1. **온보딩** — 가치 설명 + 사진/영상 접근 권한 + 알림 권한 + 알림 시간 설정
  2. **홈** — "오늘 미분류 N장", 정리 시작 버튼, 연속 정리일수(streak)
  3. **정리(스와이프)** — 큐의 항목을 하나씩, 스와이프로 배정 / 건너뛰기
  4. **앨범 선택**(모달) — 기존 앨범 리스트 + 새 앨범 생성
  5. **완료** — "정리 완료! N장" 축하 → (7일 후) 광고 1회 → 광고 제거 업셀
  6. **설정** — 알림 시간, 광고 제거 구매/복원, 프라이버시 안내
- **화면 흐름**: 온보딩 → 홈 → 정리 ↔ 앨범 선택 → 완료 → (다음 날) 홈. 홈 → 설정.
- **핵심 사용자 여정 1개 (MVP가 완성할 단 하나)**: 권한 허용 → 첫 알림 → 미분류 큐 → **사진 몇 장을 순식간에 앨범에 배정("오 편하다" = Aha moment)** → "정리 완료!" → 다음 날 재방문.

## 6. 기능 요구사항 (Functional Requirements)

| ID | 기능 | 사용자 스토리 | 우선순위 | 인수 조건 |
|----|------|---------------|----------|-----------|
| F-01 | 사진첩 접근 & 미분류 큐 생성 | 미분류분만 보고 싶다 | P0 | 권한 허용 시, 마지막 처리 이후의 신규 자산만 큐에 표시된다 |
| F-02 | 매일 알림(시간 설정) | 매일 정한 시간에 알림 | P0 | 설정한 시간에 로컬 알림이 발송된다 |
| F-03 | 스와이프 배정 UI | 빠르게 담고 싶다 | P0 | 한 손 스와이프로 항목을 앨범에 배정, 다음 항목으로 자동 이동 |
| F-04 | 기존 앨범 선택 / 새 앨범 생성 | 원하는 앨범에 담기 | P0 | 배정 중 기존 앨범 선택 또는 새 앨범 즉석 생성 가능(평면 구조) |
| F-05 | 시스템 앨범 반영 | 갤러리·백업에 반영 | P0 | iOS=앨범 태깅 / Android=폴더 이동이 실제 라이브러리에 반영된다 |
| F-06 | 처리 완료 기록(중복 방지) | 같은 걸 또 보지 않기 | P0 | 처리한 자산은 다음 큐에서 제외된다 |
| F-07 | 온보딩(권한+알림시간+첫 정리 체험) | 처음에 가치를 느끼기 | P0 | 권한 요청 → 알림 시간 설정 → 첫 정리 몇 장까지 매끄럽게 유도 |
| F-08 | 영상 지원(썸네일 + 간이 미리보기) | 영상도 정리 | P0 | 영상 썸네일 표시, 탭 시 간이 재생. 풀 플레이어·편집은 제외 |
| F-09 | 광고(완료 화면 뒤, 세션당 1회) | — | P1 | 첫 7일 이후, "정리 완료" 축하 뒤 세션당 1회만 노출. 정리 중/사진 사이엔 노출 없음 |
| F-10 | 광고 제거 인앱결제(IAP) | 광고 없이 쓰기 | P1 | 일회성 구매로 광고 영구 제거, 구매 복원 지원 |
| F-11 | 건너뛰기 / 나중에 보기 | 애매한 건 미루기 | P1 | 항목을 건너뛰면 다음 큐에 다시 등장 |
| F-12 | 분석 SDK 이벤트(리텐션 등) | (지표 측정) | P0 | North Star·KPI 산출에 필요한 이벤트가 수집된다 |
| F-13 | 프라이버시 안내 | 안심하고 쓰기 | P1 | "사진은 폰 밖으로 나가지 않음"을 온보딩·설정에서 명시 |

## 7. 비기능 요구사항 (Non-Functional)

| 항목 | 요구사항 |
|------|----------|
| 인증(로그인 방식) | 없음. 온디바이스 도구, 계정 불필요 |
| 권한·역할(RBAC) | 단일 사용자, 역할 구분 없음 |
| 과금·구독 모델 | 무료 + 광고(설치 7일 후, 완료 화면 뒤 세션당 1회) + 광고 제거 IAP(일회성). 결제는 App Store / Google Play 처리 |
| 성능 | 대량 라이브러리에서도 큐 스크롤이 부드러울 것(썸네일 캐싱·페이지네이션) |
| 보안·개인정보 | 사진·영상 **폰 밖 미전송**. 로컬 저장은 "처리한 자산 ID"뿐. 최소 수집 원칙 |
| 확장성 | 앨범/자산 수 증가 대응. 백엔드 없음 → 서버 확장 이슈 없음 |
| 사용성·접근성 | 한 손 스와이프, 큰 썸네일, 30초~2분 내 완료 가능한 흐름 |
| 지원 기기·브라우저 | iOS / Android 스마트폰. iOS PhotoKit(전체 접근), Android Scoped Storage(MediaStore) 대응 |
| 알림 | **로컬 알림**(서버 푸시 없음) |
| 제약(예산·스택·마감) | 바이브 코딩, 1인 개발, 크로스플랫폼 스택 필요. 마감 미정 |

## 8. 범위 (Scope)

- **In Scope (MVP 포함)**: F-01 ~ F-13 (사진+영상, 매일 알림, 스와이프 배정, 시스템 앨범 반영, 온보딩, 광고+광고제거 IAP, 분석, 프라이버시 안내).
- **Out of Scope (이번엔 제외)**: 삭제·중복 정리 / 태그·키워드 / AI 자동 그룹핑 / 중첩 폴더·그룹의 그룹 / 다중 앨범 배정 / 통계 리포트 / 계정·클라우드 동기화.
- **MVP 한 줄 정의**: **"하루 한 번, 미분류 사진·영상을 스와이프로 앨범에 담는다."** — 이 하나의 여정만 완벽하게.

## 9. 기술 아키텍처 스케치 (선택)

- **개략 구조**: 크로스플랫폼 앱 → 네이티브 사진 API 래핑(iOS PhotoKit / Android MediaStore) → 경량 로컬 DB(처리한 자산 ID).
- **기술 스택 후보**: Flutter 또는 React Native(크로스플랫폼·바이브 코딩 유리) / .NET MAUI(낙관의 C#·.NET 배경 활용). → `[미정: 최종 스택 선택]`
- **외부 의존성**: 광고 SDK(예: AdMob) · 인앱결제(StoreKit / Play Billing) · 분석 SDK(예: Firebase / Amplitude). 모두 서버 없이 SDK·스토어가 처리.
- **데이터 모델(개요)**: `ProcessedAsset(id, processedAt, albumId)`, `Album(로컬 참조/캐시)`. 사진·영상 원본은 저장하지 않음(참조만).
- **백엔드**: MVP에는 **불필요**.

## 10. 리스크 & 가정 (Risks & Assumptions)

| 구분 | 내용 | 대응 |
|------|------|------|
| 가정 | 타겟이 매일 알림을 "잔소리"로 느끼지 않고 습관화한다 | 알림 톤·빈도 조절, streak로 동기부여 |
| 가정 | 크리에이터의 정리 페인이 시장성이 있다(정량 미검증) | 출시 전/초기 사용자 인터뷰·설문 |
| 리스크 | iOS는 앨범=태깅이라 원본 타임라인이 안 비워짐 → "정리된 느낌"이 약할 수 있음 | 완료 화면에 진척·streak 시각화로 성취감 보완 |
| 리스크 | 광고가 정리 습관을 방해할 수 있음 | 완료(보상) 뒤에만, 세션당 1회, 첫 7일 무료 |
| 리스크 | Android는 파일 이동 시 사용자 동의 창 마찰 | 배치 승인 활용 + 이유 안내 |
| 리스크 | 대량 라이브러리에서 성능 저하 | 썸네일 캐싱·페이지네이션 |
| 리스크 | 앱의 큐 기록과 사용자 수동 정리가 어긋날 수 있음 | 기준을 "앱 처리 여부"로 명확화, 재동기화 로직 |
