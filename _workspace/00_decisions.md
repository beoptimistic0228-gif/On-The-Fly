# 확정 결정 (2026-07-02, 사용자 승인)

설계 Phase 승인 게이트에서 사용자가 확정한 결정. 이후 모든 설계·구현은 이를 따른다.

## D1. Android 정리 = 진짜 물리 폴더 이동 (배치 확정 모델)
- 정리 시 원본이 대상 폴더로 **실제 이동**(갤러리 원위치에서 사라짐).
- Android는 Scoped Storage 제약상 "한 장씩 즉시 이동" 불가 → **스테이징 → 마지막에 시스템 동의창 1회 → 배치 확정** 방식.
- 정리 UX 구조 변경: 스와이프는 "배정 예약(stage)", 세션 끝에 `commit` → `BatchAssignResult`(부분 성공 처리).
- **처리 기록은 commit 성공분에 한해**, 그리고 이동 후 최종 자산 ID로 기록(ID 재발급 대응).
- iOS는 앨범 태깅이라 즉시 반영 가능하나, **UX 일관성을 위해 동일한 stage→commit 흐름**을 사용(iOS commit은 동의창 없이 즉시 처리).

## D2. iOS/Android 제한적(limited) 접근 → 전체 접근 유도
- 제한 접근이면 "전체 미분류를 정리하려면 전체 접근이 필요"하다는 안내 화면으로 전체 접근을 유도.
- 부분 접근 정리는 MVP에서 지원하지 않음(완결감·구현 단순성 우선).

## D4. 배정 인터랙션 = 탭(앨범 칩), 스와이프 = "나중에" (2026-07-04 추인)
- PRD 5절의 "스와이프로 배정"은 앨범이 여러 개면 스와이프 하나로 특정 앨범을 못 고르는 문제가 있음(와이어프레임 v0.2 부록 A에서 제기).
- 확정: **탭(앨범 칩) = 배정 / 스와이프(좌) = 나중에(큐 재등장) / 되돌리기 = 직전 배정 취소.** 구현(sort_screen)과 일치 — 사용자 와이어프레임 문서로 추인.
- 디자인 원칙 전문은 `docs/디자인_원칙.md`.

## D5. 삭제 기능 범위 편입 (2026-07-06, 사용자 승인 — PRD 8절 Out of Scope 예외)
- 실기기 사용성 테스트에서 나온 수요 신호(로드맵 §6, `4e4c8a8`)를 근거로 사용자가 **범위 변경을 명시 승인**, 설계 착수 지시.
- 범위: **정리 루프 내 단건 삭제**에 한정. 중복정리·저장공간 비우기·일괄 삭제는 여전히 Out of Scope(Phase 1+).
- 필수 설계 검토(로드맵 §5.1 승계): (a) 파괴적 작업 — OS 표준 삭제 동의/휴지통 경유 필수, (b) 스와이프 3방향 소진 상태에서의 제스처 충돌(오조작=오삭제 방지), (c) stage→commit 모델과의 정합(삭제도 예약 후 확정인지 즉시인지), (d) 처리 ID 기록·중복방지 불변식과의 관계.
- 절차: spec-architect 설계 → platform-integrator 실현성 검토 → **사용자 설계 승인 후** 구현.
- **확정 세부 (2026-07-06 사용자 선택, 설계 §7 답변)**:
  1. **복구 모델 = 영구 삭제**(Android `createDeleteRequest`). 휴지통 미경유. 단 iOS는 플랫폼 한계로 항상 "최근 삭제됨" 30일 경유 — **플랫폼 비대칭 감수**. OS 표준 동의창이 안전장치.
  2. **삭제는 즉시 실행**(stage→commit 미적용). 삭제 탭 → OS 동의창 → 확정. 단건마다 동의창 뜨는 트레이드오프 수용. 세션 끝 commit 동의창은 이동 1회만 유지.
  3. **진입점 = 저강도 삭제 버튼**(하단 액션행 좌단, error색 축소, 최초 1회 교육 시트). 스와이프 4방향 미사용.
  4. **streak = 삭제도 인정**. 삭제 활동 기록용 로컬 테이블 추가(스키마 변경 수반). 완료 화면은 총계 + 분해 표기(추천안 채택).
  - 파생 제약: 영구+즉시라 **인앱 되돌리기(undo)는 삭제에 미적용**(OS 동의창이 유일 확인 지점). 되돌리기 버튼은 배정 예약에만 동작.
  5. **API 30 미만 삭제 버튼 미노출** (자동확정, §9 실현성 검토): minSdk=24 확인. API<30 경로는 OS 동의창 없이 직삭제되거나 예외라 안전장치 부재 → 미노출로 차단. (설계 §7-6 + §9 M2)
  6. **취소/실패 안내 통합** (자동확정, §9 Q4): 플랫폼이 취소·실패를 구분 못 함(둘 다 빈 반환) → 문구 "삭제하지 못했어요"로 통합. (§9 M3)
