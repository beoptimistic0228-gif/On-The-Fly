# 그때그때 빌드 오케스트레이터

그때그때 앱을 **설계 → 구현 → 검증** 파이프라인으로 만든다. SSOT는 `그때그때_PRD_v1.0.md`. 스택은 **Flutter** 확정.

**실행 모드: 하이브리드**
- 설계 Phase: 에이전트 팀(spec-architect 주도, platform-integrator가 실현성 역질의)
- 구현 Phase: 에이전트 팀(platform-integrator + feature-builder 협업, qa-verifier 점진 검증)

**전제:** 모든 Agent/팀원 호출은 `model: "opus"`. 중간 산출물은 `_workspace/`, 코드는 `lib/`.

## Phase 0: 컨텍스트 확인 (항상 먼저)
1. `_workspace/` 존재 여부 확인.
   - 없음 → **초기 실행** (Phase 1부터)
   - 있음 + 사용자가 부분 수정 요청(특정 화면/기능) → **부분 재실행**(해당 에이전트만 재호출, 이전 산출물 읽고 개선)
   - 있음 + 새 요구/방향 전환 → **새 실행**(`_workspace/`를 `_workspace_prev/`로 이동 후 재시작)
2. PRD와 `01_architect_*.md`(있으면)를 읽어 현재 상태 파악.

## Phase 1: 설계 (팀)
1. `TeamCreate`로 설계 팀 구성: spec-architect, platform-integrator(실현성 검토용).
2. spec-architect가 `flutter-app-design` 스킬로 4개 산출물 작성 → `_workspace/01_architect_*.md`.
3. platform-integrator가 데이터 모델·미분류 규칙의 플랫폼 실현성을 검토, 문제 시 SendMessage로 역질의.
4. **PRD에 답 없는 결정(최종 DB, 광고 트리거 등)은 사용자 승인**을 받고 확정 → plan 형태로 사용자에게 제시.
5. 설계 승인 후 팀 정리, 산출물 보존.

## Phase 2: 구현 (팀 + 점진 QA)
1. `TeamCreate`로 구현 팀: platform-integrator, feature-builder, qa-verifier.
2. `TaskCreate`로 `01_architect_tasks.md`의 태스크를 의존순서대로 할당. **핵심 루프(F-01~F-06) 우선.**
3. 데이터 흐름:
   - platform-integrator가 core 서비스(사진·DB·알림) 구현 → 공개 인터페이스를 `02_integrator_notes.md`에 문서화 → feature-builder에 SendMessage로 전달.
   - feature-builder는 그 인터페이스에만 의존해 화면 구현.
   - **각 모듈 완성 즉시** qa-verifier가 `incremental-qa`로 경계면 검증 → 이슈는 담당자에게 직접 통신.
4. 우선순위: 핵심 사전(F-01~F-08) → 온보딩(F-07) → 분석(F-12) → 광고·IAP(F-09/F-10).

## Phase 3: 통합 검증 & 종합
1. qa-verifier가 핵심 불변식(중복방지·미분류 정확성·광고 위치·프라이버시)을 최종 점검 → `03_qa_report.md`.
2. 오케스트레이터가 결과 종합, 남은 이슈·미해결 오픈이슈를 사용자에게 보고.
3. 팀 정리.

## 데이터 전달 프로토콜
- **태스크 기반**(TaskCreate): 진행·의존 관리.
- **파일 기반**(`_workspace/{phase}_{agent}_{artifact}.md`): 설계 산출물·인터페이스 계약·QA 리포트. 중간 파일 보존(감사용).
- **메시지 기반**(SendMessage): 인터페이스 전달, 역질의, 이슈 수정 요청.
- 코드는 `lib/`에 직접 작성(최종 산출물).

## 에러 핸들링
- **헛돈 에이전트(no-op) 감지 & 폴백** (중요): 서브 에이전트가 **도구 사용 0회로 반환**하거나, 지시한 산출물 파일이 실제로 생성/수정되지 않았거나, 반환이 지시 무관한 자기참조·프롬프트 에코이면 → **작업 미수행으로 간주**한다. 반환 텍스트만 믿지 말고 **실제 파일 상태(존재/변경)로 검증**하라.
  1. 1차: 더 명확·구체적인 지시로 **1회 자동 재시도**("반드시 Edit/Write로 실제 파일을 수정하라, 말로만 끝내지 마라" 명시).
  2. 재시도도 no-op이면: 작업 범위가 좁고 명확하면 **오케스트레이터가 직접 수행**(파일 읽고 편집), 넓으면 태스크를 쪼개 재할당.
  3. 어느 경우든 무엇이 헛돌았고 어떻게 처리했는지 사용자 보고에 남긴다.
- 에이전트 1회 재시도 후 재실패 → 해당 결과 없이 진행하되 보고서에 **누락 명시**.
- 플랫폼 API가 PRD 시나리오 미지원(예: Android 이동 제약) → 삭제·강행 금지, 우회안을 근거와 함께 사용자에게 올림.
- 설계·구현 상충 → 데이터 삭제 없이 출처 병기 후 사용자 판단 요청.

## 테스트 시나리오
- **정상 흐름**: 초기 실행 → 설계 4종 산출·승인 → 구현(핵심 루프 우선, 인터페이스 계약대로) → 점진 QA 통과 → 종합 보고. 기대: 배정 자산이 재로드에서 제외, 광고는 완료 뒤 1회.
- **에러 흐름**: Android 폴더 이동이 photo_manager로 제약 → platform-integrator가 우회안 2개 제시 → 오케스트레이터가 사용자 승인 요청 → 승인안 반영, QA 재검증.

## 후속 작업
- "정리 화면만 다시" 같은 부분 요청 → Phase 0에서 부분 재실행으로 분기, feature-builder만 재호출.
- 새 방향 → 새 실행. 모든 구조 변경은 CLAUDE.md 변경 이력에 기록.
